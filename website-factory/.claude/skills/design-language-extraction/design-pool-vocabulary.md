# Design Pool Vocabulary

## How this file is customised

This file documents the **design vocabulary the build agent picks from** when composing
client sites. Vocabulary = the catalogued patterns from a curated pool of top-of-niche
reference sites: hero compositions, section transitions, card grids, trust-signal
placements, gallery patterns, typography pairings, palette idioms, motion idioms.

The factory ships niche-agnostic. The **per-niche vocabulary** is generated by Module 2D
during `/build-niche-template`, which captures, scores, and synthesises 8–12 top-of-niche
sites (Apify scrape → vision score → spec gen) and writes:

```
templates/{active-niche-slug}/niche-playbook/design-vocabulary.md
```

That file follows the same structure as this stub (per-site one-liners, layout vocabulary
catalogue, typography pairings, motion idioms, etc.) but with the entries derived from
the student's chosen niche.

The build agent reads `niche-playbook/design-vocabulary.md` at Stage 10.1 (build) to
choose the visual treatment per section. The fidelity QA agent (Stage 10.4a) reads it
to verify the composed site matches the niche's design language.

If the file is missing, the agent halts with a Module-2D pointer.

## Structure of a niche design vocabulary

Every niche's `design-vocabulary.md` follows this structure:

### 1. Per-site one-liners
One sentence per reference site captured in Module 2D. Each one-liner names the dominant
palette, typography family, hero composition, and standout decorative or compositional
choice. The point is to give the build agent a vivid mental image of each reference
without re-reading the source.

### 2. Layout vocabulary catalogue
The patterns common across the niche's reference pool, grouped by section:

- **Hero compositions**, recurring composition archetypes (split-headline + form right,
  full-bleed photo with overlay headline, editorial photo collage, etc.)
- **Section-to-section transitions**, divider treatments (hard cut, diagonal slice,
  torn-paper edge, arc, topographic line)
- **Card grids**, 2-up / 3-up / 4-up / mixed-asymmetric patterns
- **Trust signal placement**, under-hero strip, footer wall, inline checkmark list,
  badge cluster
- **Service / offering grid**, photo-card with CTA, icon-tile flat, 3D rendered icon,
  list with circular bullet icons
- **Gallery**, horizontal carousel, masonry, named-project cards, map-as-gallery-stand-in

### 3. Typography pairings catalogue
Heading + body combinations observed in the reference pool, with mood notes. Format:

```
- **{heading-font} + {body-font}** ({sites using this}), {description}. Mood: {tag}.
```

Plus weight-contrast and tracking patterns observed.

### 4. Palette idioms
Recurring colour structures (warm-paper + accent; dark-monochrome + saturated accent;
near-black + cream + tan + oxblood; etc.), with frequency counts.

### 5. Motion idioms
Common motion patterns observed in the niche pool (where they motion at all): scroll-
revealed headlines, ticker marquees, parallax-on-hero-image, hover-lift on cards, etc.

### 6. Decorative motif idioms
Background patterns, corner overlays, accent shapes specific to the niche aesthetic.

### 7. Anti-patterns observed in the pool
Patterns the niche pool **avoids** that other niches do, these become explicit
"don't do this for {niche}" rules for the build agent.

## How the build agent uses this

When composing a client's site at Stage 9, the build agent:

1. Loads `niche-playbook/design-vocabulary.md` for the student's niche
2. Loads the client's `brand-dna.json` (palette, typography, motif, shape_mode)
3. Picks vocabulary entries that match the brand-dna signals (e.g. if `shape_motif:
   wave`, prefer hero compositions and dividers that incorporate fluid edges; if
   `palette: warm-paper`, prefer editorial typography pairings)
4. Composes the site by combining chosen vocabulary entries with the client's data

The vocabulary file is the bridge between the abstract "what does this niche look like"
and the concrete "what specific component patterns get rendered."

## What this file does NOT do

- It does NOT prescribe a single fixed visual treatment per niche. Every client's
  brand-dna shifts the chosen vocabulary entries.
- It does NOT replace the niche's `niche-playbook/design-vocabulary.md`. This file is the
  schema; the niche playbook is the data.
- It is NOT canonical reference for any specific niche. Module 2D writes the niche-
  specific file from the student's own research pool.
